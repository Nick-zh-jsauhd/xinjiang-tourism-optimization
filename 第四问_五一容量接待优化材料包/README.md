# 第四问 五一容量接待优化材料包

## 1. 材料包用途

本文件夹专门整理“新疆旅游线路安排”项目中第四问相关的已有材料、数据、代码快照和实验结果。第四问的核心不是为单个游客找一条路线，而是面向五一节假日客流做线路产品组合、容量分配、预约上限和动态分流。

建议汇报口径为：

> 面向五一 12 天集中客流，生成若干条可运营线路产品，并在线路容量、景区承载量和需求冲击约束下分配游客，使接待量尽可能高，同时控制热门线路过载、排队等待和极端场景下的溢出风险。

## 3. 当前已有模型

第四问目前已经形成三层模型链：

1. **线路产品容量流模型**：先构造 9 条五一线路产品，每条线路有景点集合、预计天数、12 天接待容量和吸引力评分；再把基准游客需求分配到这些线路列上。
2. **需求冲击与动态分流模型**：在需求上浮 5%、10%、20%、35% 情景下，将超载线路的游客向同区或高吸引力替代线路转移，评估剩余溢出。
3. **预约上限与鲁棒策略模型**：比较全量放票、95% 安全上限、90% 舒适上限、动态分流和分时预约策略，并在数字孪生情景中选择风险调整后更稳的策略。

数学口径可写为容量流模型：

```text
max sum_r z_r
s.t. 0 <= z_r <= Cap_r,                 for each route product r
     sum_r z_r <= D,
     spot_load_i = sum_r a_ir z_r <= Cap_i, for each bottleneck spot i
```

其中 `z_r` 表示分配到线路产品 `r` 的游客量，`Cap_r` 是线路 12 天容量，`D` 是五一目标客流，`a_ir` 表示线路是否覆盖景点 `i`。

## 4. 当前核心结果

基准容量流：

| 指标 | 当前值 |
|---|---:|
| 线路产品数 | 9 |
| 基准分配游客 | 106358 人 |
| 12 天线路总容量 | 111956 人 |
| 系统利用率 | 0.950 |
| 满载或接近满载线路 | 8 条 |

需求冲击结果：

| 需求上浮 | 请求游客 | 动态分流后接待 | 未消化溢出 | 判断 |
|---:|---:|---:|---:|---|
| 5% | 111676 | 111676 | 0 | 可由既有线路替代分流消化 |
| 10% | 116994 | 111956 | 5038 | 仅靠既有线路分流不足 |
| 20% | 127630 | 111956 | 15674 | 需要新增运力或限流 |
| 35% | 143583 | 111956 | 31627 | 需要新增线路、分时预约和前置限流 |

策略层结果：

| 策略 | 定位 |
|---|---|
| `5%` 需求上浮内 | 既有线路动态分流基本可消化 |
| `10%` 及以上需求上浮 | 必须新增运力、限流或错峰预约 |
| `staggered_prebooking` 分时预约+动态分流 | 风险感知策略选择中四类画像均选中 |
| `full_release` 全量放票 | 接待量高，但高峰与复合冲击下拥堵风险最大 |

## 6. 数据真实性边界

第四问最需要主动说明的数据边界：

- 五一宏观客流使用公开通报和文旅数据作校准，但线路级分布是模型化分配，不是真实订单数据。
- 景区容量表中部分景点有公开容量来源，部分景点是按景区类型、承载经验和安全折减参数模拟。
- 酒店房态、停车位、摆渡车运力、导游/车辆实际调度仍未接入真实 API 或平台数据。
- 当前容量流结果适合做“节假日运营策略仿真”，不能声称已经给出真实文旅部门可直接执行的排班表。

## 8. 下一轮强化方向

优先级建议：

1. 将 9 条线路产品从模板生成升级为列生成或 prize-collecting route generation。
2. 把景区容量、酒店房量、停车位、摆渡车和导游车辆统一建成多资源容量约束。
3. 加入日级/时段级预约流，而不是只看 12 天总容量。
4. 用游客偏好类型构造离散选择模型，避免需求分配只按容量和吸引力代理。
5. 建立 Q4-V2 独立求解闭环，输出脚本、报告、工作簿、审计和鲁棒策略一体化结果。
